TICKET #67373733
Switch Rollback Issues
Opened: Jan 16, 2026 Last Response: Feb 16, 2026 Closed: Mar 5, 2026
Resolved
Case Description

In the "Settings" section of the KOKOMO-S212GT-P, a message like the one shown in the attached file has started to appear. We would like to know under what circumstances this kind of message is displayed — such as when it occurs, what actions trigger it, and the possible causes behind it.

Case Summary
  • The discussion investigated why the automatic rollback mechanism did not trigger when a configuration change caused the device to lose cloud connectivity.
  • According to the design, a configuration becomes a stable configuration if the device remains online for 30 minutes. If a new configuration causes the device to lose cloud connectivity for 2 hours, the device should revert to the previous stable configuration.
  • During testing, the management VLAN was changed to VLAN 100 without DHCP or Internet access, which should have triggered the rollback, but the device remained offline instead.
  • Investigation revealed a firmware defect in the rollback script (rollback.sh), where a hard-coded service name (cloud_agent) was used instead of the variable ${CLOUD_AGENT}, preventing the stable configuration from being restored.
  • A separate issue related to SNTP configuration caused rollback log messages to fail to appear in the cloud GUI, making it harder to confirm rollback events.
  • Both issues were confirmed in the Root Cause Analysis (RCA) and are scheduled to be fixed in firmware version v1.2.125.
Ticket Details

The discussion focused on investigating why the rollback mechanism in KOKOMO switches did not function as expected when configuration changes caused the device to lose connectivity with the cloud. The issue was initially raised by the Apresia support team, who were attempting to verify the rollback feature in a test environment. According to the expected design, when Kokomo Cloud pushes a new configuration to a device and that configuration results in the device losing connection to the cloud, the system should automatically revert to the last stable configuration stored on the device.

The rollback design works in the following sequence. After a configuration change is pushed from the cloud, the device continues operating with the new settings. If the device remains connected to the cloud for at least 30 minutes without rebooting, the configuration is saved locally as a "stable configuration." However, if a new configuration is applied and the device loses connectivity to the cloud and cannot restore that connection for two hours, the device should activate a self-healing mechanism and revert to the previously saved stable configuration.

During testing, the customer changed the management VLAN on the switch to VLAN 100, which had no DHCP server, no gateway, and no Internet access. After several days of analysis, the engineering team determined that the problem was not caused by incorrect rollback conditions but by a firmware defect. Specifically, the rollback script (rollback.sh) contained a design flaw related to the cloud agent service name used during the rollback procedure. In firmware version 1.2.110, part of the rollback script used a hard-coded service name, cloud_agent, rather than dynamically referencing the service name defined by the variable ${CLOUD_AGENT}. In the Apresia environment, the cloud agent service name differs from the default name used in the script.

In addition to the rollback script issue, a separate firmware problem related to SNTP configuration was also discovered. When the SNTP server was not configured on the device, a segmentation fault could occur, which prevented rollback logs from appearing in the cloud GUI.

Both issues were confirmed by the engineering team and documented in a root cause analysis (RCA). The updated firmware corrects the rollback script so that it properly references the cloud agent service variable and ensures that the stable configuration restoration process executes correctly. It also resolves the SNTP segmentation fault problem. The Apresia support team confirmed their understanding and planned to verify the fix once firmware version 1.2.125 became available.

TICKET #76723891
OS Detection in Wireless AP
Opened: Feb 16, 2026 Last Response: Feb 24, 2026 Closed: Mar 3, 2026
Resolved
Case Description

Apresia is asking for clarification about how the wireless AP determines the operating system shown in the "OS" field under "Historical Clients." They would like to understand how the system identifies client operating systems such as Windows, Linux, Android, and others; how to handle cases where the detected OS appears to be incorrect; and whether the displayed OS can be manually changed.

Case Summary
  • The OS field in the AP Historical Clients list is determined using a mechanism similar to DHCP fingerprinting.
  • OS detection currently relies solely on DHCP Option 55 and Option 60 observed during DHCP negotiation.
  • Devices using static IP addresses or environments where DHCP packets are not observed cannot be properly identified.
  • When identification fails, the system currently classifies the device as Linux by default.
  • The detected OS cannot be manually modified through the Cloud GUI.
  • A future update to the Cloud portal and AP firmware will display unidentified devices as "Unknown" instead of "Linux".
Ticket Details

The discussion clarified how the OS field in the AP Historical Clients list is determined. Initially, Region Support explained that the AP firmware uses a mechanism similar to DHCP fingerprinting to identify the client operating system and that Linux is shown as the default when the OS cannot be identified. Region Support confirmed that the detected OS cannot be manually modified through the Cloud GUI. After further internal checks, they clarified that the fingerprinting mechanism currently relies primarily and exclusively on DHCP information, specifically DHCP Option 55 and Option 60. It was further confirmed that when DHCP is disabled, the system currently cannot detect the correct OS, and therefore such devices are shown as Linux. Region Support also stated that a future update to both the Cloud portal and AP firmware is planned so that unidentified clients will be displayed as "Unknown" instead of being incorrectly classified as Linux. The customer acknowledged the explanation and requested that the case be closed.

TICKET #74168494
About Captive-Portal
Opened: Feb 4, 2026 Last Response: Feb 11, 2026 Closed: Feb 11, 2026
Resolved
Case Description

Access is unavailable if the cloud connection fails while using these authentication methods.

Case Summary
  • Captive portal authentication methods (Click-through, KOKOMO Cloud RADIUS, and Custom RADIUS) rely on a cloud-based control plane, where the AP acts only as an executor and does not make the final authorization decision.
  • Although the splash page may be locally accessible via the walled garden, the AP must still obtain authorization from the cloud after the user clicks Continue to Internet.
  • If cloud connectivity is unavailable, the AP cannot receive the approval response and Internet access remains blocked, even in a single AP deployment.
  • Client policies are synchronized from the cloud, while authorized client information is maintained both locally and in the cloud to support roaming, VLAN grouping, and session continuity.
  • During roaming, the splash page will not be shown again if the session is still valid and cloud connectivity is maintained; it reappears only after session expiration or when devices use random MAC addresses.
  • The customer confirmed that captive portal authorization is cloud-dependent for all authentication types and acknowledged the explanation, suggesting a caution note in the manual.
Ticket Details

This conversation clarified why cloud connectivity is required for captive portal functions such as Click-through, KOKOMO Cloud RADIUS, and Custom RADIUS. Region Support first explained that these authentication methods rely on a cloud-based control plane above the access point (AP). In this design, the AP acts only as an executor and does not make the final authorization decision on its own. It was also clarified that policies are synchronized from the cloud to the APs, while authorized client information is maintained both locally and in the cloud for purposes such as roaming, VLAN grouping, and session continuity. However, the actual permission decision for captive portal login remains cloud-dependent. Regarding roaming behavior, Region Support confirmed that the splash page will not be shown again when a client moves to another AP, as long as the session is still valid and the cloud is reachable. For Custom RADIUS, Region Support explained that although WPA2/WPA3-Enterprise authentication itself can still work without cloud connectivity, the captive portal operation still depends on the cloud for authorization handling and synchronization. The customer acknowledged the explanation, stated that a caution note may be added to the manual regarding captive portal behavior during cloud disconnection, and requested closure of the case.

TICKET #71591414
AP Splash Page
Opened: Jan 28, 2026 Last Response: Feb 5, 2026 Closed: Mar 9, 2026
Resolved
Case Description

The customer asked about the default IP address or URL for the local splash page of the captive portal. An AI response suggested 192.168.1.1, but DNS lookup of falcon-jp.production.kokomocloud.com returned multiple public IP addresses.

Case Summary
  • The customer asked about the default IP address or URL for the local splash page of the captive portal. An AI response suggested 192.168.1.1, but DNS lookup of falcon-jp.production.kokomocloud.com returned multiple public IP addresses.
  • Region Support initially explained that the splash page is related to the System Reserved IP Range configured under General Settings → AP → System Reserved IP Range, and that the AP forwards the page to the client for authentication.
  • The customer questioned how the IP addresses were determined and why the resolved IP addresses were public cloud addresses instead of being within the configured reserved IP range.
  • Further clarification revealed that the captive portal splash page is delivered through HTTP redirection, not through a reverse proxy as previously suggested.
  • When a client device performs captive portal detection (e.g., accessing http://captive.apple.com, http://connectivitycheck.gstatic.com, or http://detectportal.firefox.com), the AP returns an HTTP 302 redirect to the splash login page if the detection URL is not included in the walled garden.
  • If a user attempts to access an HTTPS site directly, the AP may present its own captive portal certificate and redirect the client to the splash page, which can cause a browser security warning before the login page appears.
  • It was also confirmed that the splash page cannot be accessed directly by manually entering the URL, because the URL requires specific parameters and validation as part of the authentication process.
  • Region Support later clarified that the earlier mention of a reverse proxy was incorrect and confirmed that the actual mechanism is HTTP 302 redirection.
Ticket Details

The discussion focused on clarifying the behavior and implementation mechanism of the captive portal local splash page in KOKOMO Cloud environments. The inquiry was initially raised by the KOKOMO Support team after attempting to determine the default IP address used for the local splash page. Region Support initially responded by asking whether the inquiry referred to the Local Status Page or the Local Splash Page. After clarification, Region Support explained that the splash page is associated with the configuration setting located at General Settings → AP → System Reserved IP Range. Region Support then provided a more detailed explanation of the captive portal operation. After further clarification, Region Support corrected the earlier explanation and confirmed that the captive portal mechanism does not use a reverse proxy. Instead, the splash page is delivered using standard HTTP redirection techniques. The redirect mechanism causes the user's browser to automatically open the splash login page where authentication actions such as Continue to Internet can be performed. Region Support also explained how HTTPS traffic behaves in this scenario. Finally, Region Support acknowledged that the earlier statement describing the splash page as a reverse proxy implementation was inaccurate. The correct mechanism is the HTTP 302 redirection process used by the access point to intercept client traffic and redirect it to the captive portal login page. After receiving this clarification, the KOKOMO Support team confirmed that they understood the explanation and the discussion was concluded.

CONVERSATION #77671917
DCS Interference Monitoring Directionality
Date: Feb 24, 2026
Resolved
Highlight
  • The customer asked whether the Dynamic Channel Selection (DCS) function monitors channel traffic and noise only in certain directions relative to the AP.
  • Region Support explained that DCS monitors co-channel interference and channel utilization on the current AP channel.
  • Since KOKOMO-AP uses an omnidirectional antenna, it can detect interference from all directions around the AP.
  • Interference monitoring therefore occurs in a spherical coverage area rather than in a specific horizontal or vertical direction.
  • If a directional antenna were used, interference detection would primarily follow that antenna's radiation pattern.
  • The customer acknowledged the explanation and confirmed understanding that DCS monitoring is not directional for KOKOMO-AP.
Summary

The customer asked whether the DCS function has any directional limitations when monitoring channel traffic and noise levels. Region Support explained that DCS monitors the current AP channel for co-channel interference and channel utilization. Since KOKOMO-AP uses an omnidirectional antenna, it can receive interference from all directions in a spherical pattern. It was also clarified that if a directional antenna were used, interference would mainly be detected according to that antenna's radiation pattern. The customer acknowledged the explanation and confirmed understanding that DCS monitoring itself is not directional, and KOKOMO-AP is omnidirectional rather than directional.

CONVERSATION #77152295
Backup and Restore Compatibility Between AP Models
Date: Feb 18, 2026
Resolved
Highlight
  • The customer asked whether a backup created on a W220AX-IS could be restored to a W222BE-IS.
  • They understood that the restore function is intended to work only with the same device model and requested confirmation.

It is not a template-based mechanism and cannot be used to apply configuration across different device models. As a result, the customer's understanding was confirmed: a backup taken from one model cannot be restored to a different model, even if some settings are common between them.

CONVERSATION #215472887266060
Dashboard High CPU/Mem Alert Criteria
Date: Jan 30, 2026
Resolved
Highlight
  • The customer asked for the specific thresholds used by the KOKOMO Cloud dashboard to determine when an AP or Switch is experiencing high CPU or memory load.
  • Region Support confirmed that a High CPU alert is triggered when the average CPU utilization exceeds 90% over a 5-minute period.
  • For memory usage, there is currently no defined alert threshold used by the dashboard.
  • Memory utilization is instead monitored internally through software safeguards and service watchdog mechanisms to ensure operation within safe limits.
  • If abnormal memory behavior such as a memory leak occurs, the issue is typically investigated by R&D and resolved through firmware updates rather than configuration changes.
  • Therefore, the dashboard label High CPU/Mem AP is effectively triggered by the CPU utilization condition only.
CONVERSATION #215472887301373
High Channel Utilization Detection Criteria
Date: Jan 30, 2026
Resolved
Highlight
  • The customer asked about the criteria used by the KOKOMO Cloud dashboard to determine when channel utilization is considered high.
  • Region Support explained that High Channel Utilization is detected when the AP's operating channel experiences 70% interference or channel busy time continuously over 15 minutes.
  • This condition is typically indicated on the AP list page, where channel utilization information can be observed.
  • By hovering over the channel utilization graph, users can view the possible cause and recommended actions.
  • In environments with many 2.4 GHz devices (e.g., offices with BLE devices, wireless mice, or keyboards), high utilization may be expected and can sometimes be ignored.
  • The customer acknowledged the explanation and confirmed that the case could be closed.

INTERCOM CHAT #215473239160749
ESG620 Diag Tools Issue
Date: Feb 25, 2026
Resolved
Highlight
  • The customer reported that Diag Tools on the ESG620 gateway could not be used in a test environment, while the same function remained available on the AP.
  • The test topology consisted only of the gateway and AP connected to the internal network, with communication to Google DNS blocked.
  • Region Support explained that the ESG620 determines Internet availability using ICMP reachability and DNS resolution to Google DNS servers (8.8.8.8 and 8.8.4.4).
  • The Diag Tools function on the gateway depends on this Internet connectivity check; if the check fails, the feature remains unavailable even though the device may still appear online in the cloud.
  • In contrast, APs and switches verify connectivity by pinging the local gateway, allowing their Diag Tools to function even when Google DNS is unreachable.
