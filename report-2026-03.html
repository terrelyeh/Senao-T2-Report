TICKET #82514414
Summary of RADIUS Test Failure (KOKOMOCloud / APRESIA)
Opened: Mar 12, 2026 Last Response: Mar 30, 2026 Closed: Apr 2, 2026
Resolved
Case Description

When testing the KOKOMOCloud feature while using Custom RADIUS under WPA3-Enterprise, the customer reports that an error stating a certificate is missing is displayed, even though RadSec is disabled.

The same error is also observed when performing the test from APRESIA. Could you please advise if there is a way to perform the test without a certificate, or if there are any required settings to bypass this requirement?

Case Summary
  • The Phenomenon: When performing a RADIUS test under WPA3-Enterprise, a "certificate missing" error was displayed even if RadSec was disabled. Actual client authentication remained functional; only the cloud-based test tool failed.
  • Root Cause (AP Defect): The issue was identified as a protocol mismatch in the AP's diagnostic script. The script was hardcoded to use GTC (Generic Token Card), while the customer's RADIUS server required MS-CHAPv2 (the industry standard).
  • Misleading Error Logic: The system used a generic error message regarding RadSec certificates for any RADIUS connection failure, leading to confusion when RadSec was not actually in use.
  • Immediate Resolution: R&D applied a manual fix to the affected AP via SSH, updating the test script to support both MS-CHAPv2 and GTC.
  • Long-term Corrective Actions:
    • Firmware Update: The fix will be integrated into the next general firmware release to ensure native compatibility for all users.
    • UI Enhancement: The cloud error message is being updated to accurately reflect connection failures versus certificate issues (ETA: Late April 2026).
Ticket Details

In March 2026, a significant technical discrepancy was identified within the KOKOMOCloud and APRESIA monitoring platforms concerning the KOKOMO-W220AX-IS Access Point (AP). The primary issue involved a persistent "certificate missing" error displayed during the RADIUS Test function, specifically when utilizing WPA3-Enterprise security. This error appeared even when the RadSec (RADIUS over TLS) feature was disabled, leading to significant confusion regarding whether a digital certificate was mandatory for standard authentication testing.

Following an escalation to EnGenius R&D, the investigation revealed that the failure was not caused by a misconfiguration of the customer's RADIUS server or a lack of certificates. Instead, the defect was located within the AP's internal diagnostic script. The script was hardcoded to initiate authentication using the GTC (Generic Token Card) inner authentication method. However, the customer's RADIUS environment strictly enforced the MS-CHAPv2 protocol, which is the current industry standard. Because the diagnostic tool could not negotiate the correct protocol, the authentication failed.

Furthermore, the Cloud UI utilized a generic error logic: any failure in the RADIUS handshake triggered a default message stating that the RadSec certificate was missing or invalid. This created a false diagnostic path, as the actual cause was a protocol mismatch rather than a security credential issue.

The R&D team provided an immediate resolution by manually patching the affected AP via SSH, updating the diagnostic script to support both MS-CHAPv2 and GTC. This successfully restored the functionality of the Cloud Test tool. For broader resolution, a permanent fix is scheduled for the next general firmware release. Additionally, the Cloud team is implementing a UI patch (ETA: April-End 2026) to provide dynamic, context-aware error messages that distinguish between protocol mismatches and actual certificate failures.

Regarding Language Settings and Procedural Oversights

During the investigation, the Engineering team temporarily switched the interface language to English to ensure operational accuracy and prevent potential misinterpretations of the Japanese UI.

We acknowledge that any action capable of triggering a configuration application should be performed only after explicit customer confirmation, particularly when no configuration changes have been requested. In this instance, this protocol was overlooked. To prevent recurrence, we have reinforced our internal support procedures. Moving forward, all investigative activities in customer environments will strictly require prior authorization for any action that may impact device configurations. We are ensuring all engineers are retrained on this mandatory protocol.

TICKET #83633969
"Web Authentication Failed" Error
Opened: Mar 16, 2026 Last Response: Mar 19, 2026 Closed: Apr 2, 2026
Resolved
Case Description

User Account: so-aoi@iij.ad.jp  ·  Incident Period: March 11 to March 16, 2026

The user reported that even after entering the email address and password on the login screen, the error "web authentication failed" was displayed and they were unable to log in from March 11 to March 16. Currently, the issue has resolved itself and the user is now able to log in. However, we would like to know the cause of the issue and the reason for the "web authentication failed" error.

During the time the issue occurred, the user attempted the following actions but was still unable to log in:

  • Attempted login using different browsers (Microsoft Edge and Google Chrome)
  • Tried using a private window and cleared cache and cookies
  • Changed the password
Case Summary
  • Root Cause (Google reCAPTCHA): The "web authentication failed" error is uniquely mapped to the Google reCAPTCHA v3 verification mechanism introduced in February 2026. The login attempts were blocked because the system assigned a risk score below the required security threshold.
  • Identification of Risk Factors: Even though the user is a legitimate human, the low security score (bot-like classification) was likely triggered by:
    • Network Environment: A low IP reputation, use of a VPN, proxy servers, or an enterprise network previously associated with suspicious traffic.
    • Browser Environment: Automation plugins, aggressive ad blockers, or incomplete browser fingerprinting.
    • Behavioral Analysis: Mechanical patterns in mouse movements or a lack of normal browsing/cookie history.
  • Reason for Persistent Failure: The issue persisted despite changing browsers because reCAPTCHA evaluates the network path (IP level) and device characteristics.
  • Resolution over Time: The issue resolved "naturally" because reCAPTCHA scores are dynamic. As the IP reputation improved over several days, the score eventually rose above the threshold.
  • Distinction from Credential Errors: The system follows a different code flow for incorrect passwords. The "web authentication failed" error occurs before the 2FA step, confirming it is a security gate failure.
  • Recommended Mitigation:
    • If the error recurs, users should try switching networks (e.g., using a mobile hotspot) to immediately change the IP-based risk evaluation.
    • Support can evaluate implementing a manual whitelist mechanism if specific legitimate users are frequently blocked.
Ticket Details

Between March 11 and March 16, 2026, a persistent login issue was reported for the user account so-aoi@iij.ad.jp. Despite entering valid credentials, the user repeatedly encountered a "web authentication failed" error. The failure occurred prior to the Two-Factor Authentication (2FA) stage and persisted across multiple browsers and password resets, eventually resolving without manual intervention.

Following a formal investigation by the EnGenius and Apresia support teams, it was determined that the error is uniquely associated with the Google reCAPTCHA v3 security mechanism, which was officially integrated into the login architecture in February 2026. Unlike a standard credential mismatch (which triggers a "Mail Or Password Is Incorrect" message after the 2FA step), the "web authentication failed" error indicates that the login attempt was blocked because the reCAPTCHA risk score fell below the system's security threshold.

The reCAPTCHA engine evaluates login environments based on a variety of data points. Even for legitimate human users, a low (bot-like) score can be triggered by network reputation, browser environment factors (ad-blockers, automation-related plugins), and behavioral patterns (mouse movements, click cadence).

The issue resolved naturally as the user's IP reputation improved within Google's dynamic scoring database or the network context changed. To mitigate future occurrences, users are advised to switch network environments (e.g., using a mobile hotspot) to bypass IP-specific blocks. Additionally, the development team is evaluating a manual whitelist mechanism to support legitimate users who are consistently misidentified by the automated security gate.

TICKET #86120230
Unable to Use Activity and Internet Connectivity in Diagnostic Tools
Opened: Mar 26, 2026 Last Response: Mar 31, 2026 Closed: Mar 31, 2026
Resolved
Case Description

When accessing the "Activity" and "Internet Connectivity" pages from the diagnostic menu, the dialog window itself opens, but the content never loads. The screen remains stuck with a blue loading animation spinning indefinitely.

This behavior has been present since the initial installation, and the pages have never displayed successfully when accessed from our corporate PC environment. Meanwhile, other diagnostic tools such as Cable Test and Traceroute work normally and display their results without any issues.

During the investigation, the following was confirmed:

  • The issue occurs only when accessing the interface from the corporate network environment.
  • When directly connecting the same PC to the internet (bypassing the corporate proxy), the issue disappears, and both "Activity" and "Internet Connectivity" load correctly.
Case Summary
  • Issue: "Activity" and "Internet Connectivity" pages in diagnostic tools fail to load, remaining stuck on a continuous loading screen.
  • Environment: Issue occurs only within a corporate network using a proxy with TLS/SSL inspection; functions normally when bypassing the proxy.
  • Scope: Other diagnostic tools (e.g., Cable Test, Traceroute) operate normally, indicating a feature-specific issue.
  • Root Cause: The affected pages rely on persistent WebSocket (WSS over port 443) connections for real-time data streaming.
  • Technical Constraint: TLS/SSL inspection by the corporate proxy disrupts the WebSocket handshake (HTTP Upgrade), blocking data transmission.
  • Impact: Without the WebSocket connection, real-time data cannot be retrieved, resulting in indefinite loading behavior.
  • Resolution: Exclude relevant domains (e.g., dolphin-jp-*.production.kokomocloud.com) from TLS/SSL inspection or proxy filtering.
  • Outcome: Issue explanation provided to the customer; ticket successfully closed after guidance on proxy configuration.
Ticket Details

This report outlines an issue encountered when accessing the "Activity" and "Internet Connectivity" pages within the diagnostic tools of a cloud-managed device. While the dialog interface opens as expected, the content fails to load and remains stuck in a continuous loading state. This behavior has been consistently observed since the initial installation and occurs exclusively within a corporate network environment.

Further investigation revealed that the issue does not occur when the same client device is connected directly to the internet, bypassing the corporate proxy. In such cases, both diagnostic pages load correctly and display real-time data as intended. Additionally, other diagnostic tools, such as Cable Test and Traceroute, appear to function normally under both network conditions, suggesting that the problem is isolated to specific features.

The root cause was identified as the reliance of the "Activity" and "Internet Connectivity" pages on persistent WebSocket (WSS) connections over port 443 to stream real-time data. In corporate environments where TLS/SSL inspection is enabled, proxy servers intercept and decrypt HTTPS traffic for security purposes. This inspection process disrupts the WebSocket connection establishment, particularly the HTTP Upgrade handshake required for WSS communication.

As a result, the affected pages are unable to receive data streams, leading to indefinite loading behavior. To resolve this issue, it is recommended to configure the corporate proxy to bypass TLS/SSL inspection for the relevant domains, such as dolphin-jp-*.production.kokomocloud.com. Implementing this exclusion allows WebSocket connections to be established successfully, restoring full functionality of the diagnostic features.
